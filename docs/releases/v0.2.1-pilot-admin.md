# TrustOps v0.2.1-pilot-admin Release Notes

**Release date:** 2026-06-18

## Overview

TrustOps v0.2.1-pilot-admin makes the platform **easier, safer, and more credible to operate during real MDR/SOCaaS pilots**. Building on v0.2.0 operational pilot features, this release focuses on pilot administration, integration key lifecycle, client-ready reporting, controlled evidence handling, and lightweight ITSM export stubs.

## What Changed

| Area | Summary |
|------|---------|
| Integration keys | Per-client hashed keys with rotation/revocation |
| Admin console | Full Pilot Admin Console v2 in the UI |
| Report branding | Org and client-level cover page branding |
| Evidence | Controlled file uploads with visibility rules |
| External tickets | ServiceNow/Jira summary export stubs |
| Version | `0.2.1-pilot-admin` across API, env, and docs |

## Pilot Administration

- **Pilot Admin Console v2** at `/app/admin` with dashboard, clients, users, integration keys, report branding, pilot checklist, and demo reset
- **Backend APIs:** `GET /admin/summary`, `GET /admin/pilot-checklist`, `POST /admin/demo-reset`
- **Pilot checklist** tracks 11 setup items with Complete / Needs attention / Not started status
- **Demo reset** available only in `local-demo` deployment mode

## Integration Key Management

- New `integration_keys` table — never stores raw keys
- Create, rotate, revoke, and disable per client
- Raw key returned **once** at creation/rotation
- SHA-256 hash with application pepper
- Sentinel and webhook ingestion try per-client key first; env key fallback in `local-demo` only
- `last_used_at` updated on successful ingestion
- All key lifecycle events audited

## Report Branding

- Organization-level and client-specific branding
- Provider name/logo, client logo, report title, prepared by/for, confidentiality footer
- Cover page on report preview and print/PDF export
- Suggested export filename: `TrustOps_{ClientName}_{YYYY-MM}_SOC_Value_Report.pdf`

## Evidence Uploads

- Upload endpoint: `POST /cases/{case_id}/evidence/upload`
- Download: `GET /cases/{case_id}/evidence/{evidence_id}/download`
- Allowed types: `.txt`, `.log`, `.csv`, `.json`, `.png`, `.jpg`, `.jpeg`, `.pdf`
- Default size limit: 10 MB (`MAX_EVIDENCE_FILE_MB`)
- Visibility: Internal (SOC only) or Client Visible
- Local-demo storage: `backend/storage/evidence/`

## ServiceNow/Jira Export

- `GET /cases/{case_id}/external-ticket-summary?target=servicenow|jira|generic`
- `POST /cases/{case_id}/external-ticket-link` — save external ticket reference
- Case workspace External Ticket section with copy-to-clipboard
- **No bidirectional sync** — export stubs only

## Security Notes

- Raw integration keys never persisted
- Client users cannot access admin APIs, internal evidence, QA notes, or raw AI prompts
- Upload/download actions write audit log entries
- Cross-client key ingestion blocked
- Revoked/disabled/rotated keys reject ingestion

## Validation Results

```bash
cd backend && pytest tests/ -v          # 69 passed
API_URL=http://localhost:8001 python scripts/validate_demo.py
cd frontend && npm run build:check
```

`validate_demo.py` verifies version, integration keys, pilot checklist, branding, evidence upload, external ticket summary, admin access control, and revoked key rejection.

## Known Limitations

- No full ServiceNow/Jira bidirectional sync
- Evidence storage is local filesystem in local-demo; production should use object storage (Azure Blob, S3)
- No SSO/OIDC yet
- No SOC 2 certification yet
- No automated containment actions by design

## Upgrade Notes

1. Set `APP_VERSION=0.2.1-pilot-admin` in `.env`
2. Run `cd backend && alembic upgrade head` (migration `003_pilot_admin`)
3. Create per-client integration keys via Pilot Admin → Integration Keys
4. In pilot/production modes, retire shared `WEBHOOK_API_KEY` for client-specific keys
5. Optional: configure `MAX_EVIDENCE_FILE_MB` and evidence storage backend for pilot deployments
